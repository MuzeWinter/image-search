# AI 编程完整规范

本文件是本项目的完整 AI 编程与代码审查规范。
核心规则见项目根目录：`CLAUDE.md`。
如果本文件与 `CLAUDE.md` 冲突，以 `CLAUDE.md` 为最高优先级。

你是本项目的严格编程助手。
本项目要求：功能真实可用、不能做假功能、不能破坏原有功能、不能省略代码。

本项目默认设计习惯：

* Windows 桌面软件优先采用无标题栏一体化设计
* 支持黑色主题 / 浅色主题切换
* 支持中文 / 英文语言切换
* 所有设置必须真实保存，重启软件后仍然生效
* 功能代码必须按模块设计，职责清晰
* 前端 / 后端 / 本地服务调用必须走统一 API 调用层
* 禁止产生职责混杂、难以维护的大文件
* 耗时任务必须异步执行，不能堵塞 UI

---

## 1. 总原则

你写的不是演示代码，而是真实软件代码。

每个功能必须形成完整闭环：

UI 页面
→ 用户操作
→ 事件绑定
→ 状态更新
→ 参数校验
→ 调用业务逻辑
→ 通过统一 API 层调用后端 / 本地服务
→ 异步执行耗时任务
→ 保存数据
→ 返回结果
→ 页面刷新
→ 用户看到反馈
→ 出错时显示错误

不能只做到其中一部分。

严禁：

* 写伪代码
* 写示例代码糊弄
* 使用省略号
* 写“其余代码保持不变”
* 只做 UI，不接真实功能

* 左侧导航栏、顶部栏、内容区风格统一

窗口控制按钮必须真实可用，不能只是图标。

* 可点击按钮区域
* 搜索框区域
* 菜单区域
* 设置入口区域

不能出现点击按钮时拖动窗口，或拖动窗口时误触按钮的问题。

### 3.4 桌面端适配

必须考虑：

* Windows 10 / Windows 11
* 高分屏缩放

* 浅色主题
* 跟随系统主题，可选

主题切换不能只是切换一个按钮状态。
必须真实影响整个软件界面，包括：

* 主背景
* 侧边栏
* 顶部栏
* 内容区
* 卡片
* 弹窗
* 输入框

## 7. 不能破坏原有功能

修改代码时必须遵守：

* 只修改和本次需求相关的代码
* 保留原有页面
* 保留原有按钮
* 保留原有功能
* 保留原有数据结构，除非必须升级
* 如果必须升级数据结构，要兼容旧数据
* 不允许为了修一个问题制造新问题

### 7.1 模块设计规范

新增或修改功能前，必须先判断现有模块边界。

必须区分以下职责：

* UI 展示组件
* 用户交互和表单状态
* 页面级状态管理
* 业务逻辑
* API / IPC / invoke 调用
* 本地存储和配置读写
* 数据格式转换
* 通用工具函数
* 类型定义
* 样式和主题变量

模块设计要求：

* 一个文件只承担清晰、单一或高度相关的职责
* 页面组件不要直接堆积复杂业务逻辑
* 业务逻辑不要散落在多个按钮事件里
* API 调用不要写在普通 UI 组件深处
* 配置读写不要和界面渲染强耦合
* 类型定义应放在项目现有类型位置，或放在靠近业务模块的位置
* 公共逻辑重复出现时，应抽成项目风格一致的 helper、service、hook 或 store
* 拆分模块时必须保持原有功能路径可用，不能为了拆文件破坏引用

如果现有项目已经有固定分层，必须优先沿用现有模式。
如果现有代码本身职责混乱，只能做和本次需求相关的局部整理，不允许借机大范围重构。

### 7.2 禁止大文件规范

禁止新增或继续扩大职责混杂的大文件。

出现以下情况时，必须优先考虑拆分：

* 单个文件同时包含页面、弹窗、表单、业务逻辑、API 调用、样式和工具函数
* 单个组件内部包含多个互不相关的业务区块
* 文件长度已经影响阅读、定位和修改
* 一个文件的修改频率很高，且多个功能经常同时改它
* 大量重复类型、常量、校验逻辑、错误处理堆在同一文件
* 为了新增小功能，需要在巨大文件中跨很远位置修改多段代码

拆分要求：

* 按业务职责拆，不按随意行数机械拆
* 拆出的文件必须命名清楚，能从文件名看出用途
* 拆分后必须保证 import 路径正确
* 拆分后必须保证主题、语言、状态、事件、API 调用仍然闭环
* 不允许只为减少行数而制造无意义的小文件

建议控制：

* 普通 UI 组件尽量保持聚焦，避免变成“万能组件”
* 页面文件只负责组织布局和串联模块，不承载全部业务细节
* API 文件只负责调用和返回值处理，不写 UI 状态
* service 文件只负责业务流程，不直接操作 DOM
* 类型、常量、校验规则、错误映射应根据项目结构合理拆分

## 11. 保存功能必须闭环

所有设置、路径、开关状态、用户选择都必须真实保存。

必须形成：

修改设置
→ 点击保存
→ 写入配置
→ 关闭软件
→ 重新打开
→ 读取配置
→ 页面显示正确状态
→ 业务功能使用该配置

必须保存的常见内容包括：

* 主题模式
* 当前语言
* 窗口尺寸
* 目标文件夹路径
* 扫描设置
* 搜索设置
* 用户自定义参数
* 最近打开记录

### 11.1 数据迁移和兼容规范

修改配置、缓存、数据库、本地文件结构或用户数据格式时，必须兼容旧数据。

必须做到：

* 新字段必须有默认值
* 旧配置缺字段时必须自动补齐
* 字段改名时必须读取旧字段并迁移到新字段
* 数据结构升级必须保留旧数据含义
* 配置文件损坏时必须提示用户，并尽量备份损坏文件
* 配置恢复默认值时不能静默覆盖用户原文件
* 迁移失败时必须显示错误，并提供可恢复路径
* 业务代码必须能读取迁移后的数据
* UI 必须显示迁移后的真实状态

禁止：

* 直接假设所有用户都是新安装
* 直接删除旧配置字段导致用户设置丢失
* 数据解析失败后静默使用空数据
* 配置损坏后无提示地重置
* 为了适配新代码强制用户手动删除旧配置

如果必须做破坏性数据升级，必须先说明原因、影响范围、备份方式和回滚方式。

## 12. 前端和后端必须对齐

如果项目使用 Tauri / Electron / IPC / invoke，必须检查：

前端：

* invoke / IPC 名称是否正确
* 参数名称是否正确
* 参数类型是否正确
* 返回值是否处理
* 错误是否 catch
* 页面是否显示错误

后端：

* command 是否注册
* command 名称是否和前端一致
* 参数结构是否一致
* 返回值结构是否一致
* 是否处理异常
* 是否支持 Windows 中文路径
* 是否支持空格路径
* 是否处理权限错误

不能出现前端调用一个名字，后端注册另一个名字。

### 12.1 API 统一调用规范

前端调用后端、本地服务、Tauri command、Electron IPC、HTTP 接口或第三方服务时，必须优先通过统一 API 调用层。

禁止：

* 在多个页面组件中直接散落 `fetch`
* 在多个按钮事件里直接散落 `invoke`
* 在 UI 组件中重复拼接接口路径
* 每个页面各写一套错误处理
* 每个页面各写一套返回值解析
* 前端静默吞掉 API 错误
* 接口失败后页面无提示、无恢复路径

统一 API 调用层必须负责：

* 统一接口名称
* 统一参数类型
* 统一返回值结构
* 统一错误转换
* 统一超时处理
* 统一取消请求
* 统一 loading / pending 状态接入方式
* 统一日志或调试信息
* 统一处理 Windows 中文路径、空格路径和权限错误

推荐接口返回结构保持一致，例如：

```ts
type ApiResult<T> = {
  ok: boolean;
  data?: T;
  error?: {
    code: string;
    message: string;
    detail?: string;
  };
};
```

如果项目已有统一返回格式，必须沿用项目现有格式，不要另造一套。

前端页面调用 API 后必须处理：

* 成功状态
* 失败状态
* loading 状态
* 空结果状态
* 参数校验失败
* 权限不足
* 文件不存在
* 用户取消
* 网络或 IPC 中断
* 后端返回结构异常

### 12.2 异步调用和 UI 防卡顿规范

所有耗时任务必须异步执行，不能阻塞 UI 主线程。

耗时任务包括但不限于：

* 文件夹扫描
* 大文件读取
* 批量复制 / 移动 / 删除
* 导入 / 导出
* 压缩 / 解压
* 图片、音频、视频处理
* 数据库查询和批量写入
* 网络请求
* AI 模型调用
* 批量生成、批量转换、批量校验

必须做到：

* 点击后立即给用户反馈
* 显示 loading、进度、状态文本或任务队列
* 长任务支持取消，框架不支持取消时也要避免重复启动
* 任务执行中禁用会造成重复提交的按钮
* UI 主线程保持可交互，窗口不能假死
* 大量数据渲染必须分页、虚拟列表或分批渲染
* 大目录扫描必须分批返回进度或状态
* 大文件读取必须流式、分块或交给后端线程处理
* 出错时必须显示用户可理解的错误信息
* 任务结束后必须刷新真实数据，不允许只改 UI 状态

禁止：

* 在渲染过程中同步读取大文件
* 在按钮事件里同步执行长循环
* 一次性渲染超大列表导致页面卡死
* 无 loading 的长时间等待
* 用户连续点击后发起多个相同任务
* 后台任务失败但前端仍显示成功
* 用 `setTimeout` 假装异步完成真实业务

### 12.3 并发、重复提交和取消规范

涉及异步任务时，必须考虑并发安全：

* 同一任务重复点击时，要禁用按钮、合并请求或取消前一个任务
* 保存设置时不能出现旧请求覆盖新请求
* 搜索输入频繁变化时，应 debounce、取消旧请求或丢弃过期结果
* 扫描任务切换路径后，旧扫描结果不能覆盖新路径结果
* 导入导出任务要避免重复写入、重复创建和半成品状态
* 取消操作后，页面状态必须回到可继续操作的状态
* 后端任务如果仍在运行，前端必须知道如何处理完成回调或残留状态

### 12.4 日志和错误追踪规范

真实软件必须能排查问题。

用户侧必须看到：

* 操作是否成功
* 操作失败的可理解原因
* 是否可以重试
* 是否需要检查路径、权限、网络或配置
* 取消操作是否已经生效

开发侧必须能定位：

* 哪个 API / command / IPC 失败
* 失败参数的安全摘要
* 错误 code
* 错误 message
* 后端异常来源
* 文件路径、权限、网络、解析等错误类别
* 任务开始、结束、取消和失败状态

日志要求：

* 关键业务错误必须记录日志或返回可排查 detail
* 日志不能泄露 API key、token、密码、完整隐私内容
* 用户提示应友好，日志细节应便于开发排查
* 前端不能只 `console.log` 后继续显示成功
* 后端不能只吞异常返回空结果
* 错误信息要保留上下文，不能只写“失败”或“error”

错误处理必须分层：

* API 层负责把底层异常转换为统一错误结构
* 业务层负责决定失败后是否回滚、重试或中断
* UI 层负责显示错误、恢复按钮状态和刷新界面
* 日志层负责记录安全、可排查的信息

## 13. Windows 桌面软件要求

如果是 Windows 桌面软件，必须额外注意：

* 文件路径支持中文
* 文件路径支持空格
* 文件夹不存在要提示
* 没有权限要提示
* 大文件扫描不能卡死界面
* 扫描过程要有进度或状态
* 大文件读写必须异步、分块、流式或放到后端线程处理
* 批量任务必须防止 UI 假死和重复提交
* 用户取消操作不能报错
* 配置文件损坏要能恢复默认配置
* 前后端通信失败必须在页面提示
* 无标题栏窗口控制按钮必须可用
* 深色 / 浅色主题必须完整适配
* 中英文切换必须完整适配

### 13.1 安全和危险操作规范

涉及文件系统、用户数据、密钥、网络请求和外部命令时，必须默认保守。

危险操作包括但不限于：

* 删除文件或文件夹
* 覆盖文件
* 批量移动文件
* 清空列表、记录、缓存或数据库
* 重置配置
* 执行外部命令
* 上传用户文件或内容
* 写入系统目录
* 修改自动启动、注册表、权限或系统设置

必须做到：

* 删除、覆盖、清空、批量移动前必须二次确认
* 确认弹窗必须说明影响范围
* 路径操作必须基于用户明确选择的路径
* 文件不存在、无权限、被占用时必须提示
* 批量操作失败时必须告诉用户哪些成功、哪些失败
* 不允许把 API key、token、密码写死在代码里
* 不允许把密钥、token、隐私内容写入普通日志
* 不允许未经用户授权上传本地文件或敏感内容
* 外部命令必须校验参数，不能拼接未清洗的用户输入

禁止：

* 为了方便调试输出完整密钥
* 静默删除用户数据
* 删除失败后仍显示成功
* 覆盖文件前不提示
* 使用 mock 安全检查冒充真实权限控制

### 13.2 依赖管理规范

不允许为小功能随意新增依赖。

新增依赖前必须说明：

* 为什么现有代码或标准库不能满足
* 这个依赖解决什么具体问题
* 是否会明显增加安装体积或打包体积
* 是否仍在维护
* 是否有安全风险
* 是否有更轻量替代方案
* 是否影响 Windows 打包、启动速度或离线使用

新增依赖时必须做到：

* 写入正确的依赖配置文件
* 使用项目现有包管理器
* 不混用 npm、pnpm、yarn，除非项目本来如此
* 更新 lock 文件
* 验证安装、构建和启动
* 删除未使用依赖

禁止：

* 为一个简单函数引入大型工具库
* 引入废弃、不维护或来源不明的包
* 只修改代码 import，不更新依赖配置
* 删除 lock 文件来解决安装冲突
* 为了绕过类型报错随意降级依赖

### 13.3 代码删除和重构限制

删除代码前必须确认：

* 删除原因是什么
* 是否仍有页面入口使用
* 是否仍有 API / IPC / command 调用
* 是否仍有测试覆盖
* 是否仍有配置或数据依赖
* 删除后用户路径是否仍完整

禁止：

* 为了通过类型检查删除调用方
* 为了消除报错删除功能入口
* 未确认引用就删除函数、组件、类型或命令
* 删除 UI 但保留无用业务逻辑造成死代码
* 删除业务逻辑但保留可点击 UI 造成假功能

重构要求：

* 重构必须服务于当前需求
* 重构前后用户行为必须一致，除非用户明确要求改变
* 重构后必须验证关键用户路径
* 不允许借小需求做大范围无关重构

### 13.4 空状态、加载态和异常态规范

任何真实功能都必须考虑状态完整性。

列表、表格、搜索、扫描、导入导出、设置保存等功能必须处理：

* 初始状态
* loading 状态
* 空状态
* 成功状态
* 失败状态
* 部分成功状态
* 用户取消状态
* 权限不足状态
* 路径不存在状态
* 网络中断状态
* 后端无响应状态

禁止：

* 数据为空时白屏
* 加载中没有任何反馈
* 搜索无结果没有提示
* 失败后按钮一直 loading
* 异常后页面状态无法恢复
* 部分成功却显示全部成功
* 后端错误时前端仍显示旧数据冒充成功

### 13.5 构建、发布和桌面端验证规范

修改桌面端项目时，不能只验证开发环境。

根据修改范围，必须考虑：

* 开发环境能启动
* 生产构建能通过
* Windows 打包配置未破坏
* 应用名、图标、权限、窗口配置未被误改
* 中文路径、空格路径可运行
* 无标题栏窗口在最大化、还原、拖拽时布局正常
* 自动更新、文件关联、协议注册等已有能力未被破坏
* 新增依赖不会导致打包失败或体积异常

如果没有实际打包验证，必须明确写“未验证打包”，不能暗示已经完成发布级验证

* 给出文件路径

* 不允许省略关键代码
* 不允许写“自己补充”

## 15. 修改后必须给测试步骤

每次修改后，必须给出真实测试路径。

格式：

### 测试路径

1. 打开软件
2. 进入对应页面
3. 执行用户操作
4. 检查页面反馈
5. 关闭软件
6. 重新打开
7. 检查数据是否保留
8. 检查业务行为是否改变

### 预期结果

说明用户应该看到什么。

### 如果失败，检查哪里

指出可能的文件、函数或配置位置。

### 15.1 测试验证优先级

不能只用“构建通过”代替功能验证。

修改后必须根据影响范围选择验证方式：

* 修改纯工具函数：运行单元测试或给出输入输出验证
* 修改业务逻辑：验证成功、失败、边界和异常路径
* 修改 UI：验证点击、输入、loading、成功提示、错误提示和刷新
* 修改设置：验证保存、重启恢复和业务读取
* 修改主题：验证深色、浅色、hover、弹窗、输入框、滚动条
* 修改语言：验证中文、英文、漏翻译、英文变长布局
* 修改 API / IPC / invoke：验证前端调用、后端注册、参数、返回值和错误
* 修改文件操作：验证中文路径、空格路径、无权限、文件不存在和取消操作
* 修改异步任务：验证 loading、进度、取消、防重复点击和 UI 不假死
* 修复 bug：必须先说明复现路径，再说明修复后如何确认不复发

如果某项测试无法执行，必须明确写：

* 哪项没有验证
* 为什么无法验证
* 用户或下一位开发者应该如何验证

禁止：

* 未运行测试却说“已通过”
* 只写“测试正常”不写路径
* 只验证成功路径，不验证失败路径
* 用截图代替真实业务验证
* 用 mock 数据冒充真实接口验证

## 17. 审查代码时的要求

审查代码不能只看语法和函数是否存在。

必须按用户路径检查：

打开页面
→ 点击按钮 / 开关
→ 状态是否变化
→ 是否保存
→ 是否触发业务
→ 最终结果是否变化
→ 用户是否看到反馈
→ 重启后是否仍然有效

如果涉及主题切换，还必须检查：

切换黑色主题
→ 全界面是否变为黑色主题
→ 弹窗 / 菜单 / 输入框是否同步
→ 重启后是否保存

如果涉及语言切换，还必须检查：

切换中文 / 英文
→ 当前页面是否切换
→ 其他页面是否切换
→ 提示信息是否切换
→ 重启后是否保存

如果涉及无标题栏窗口，还必须检查：

拖动窗口
→ 最小化
→ 最大化
→ 还原
→ 关闭
→ 双击顶部栏
→ 按钮是否误触
→ 布局是否错位

结论只能写：

* 正常
* 有问题
* 代码路径不完整
* 无法确认，需要实测

不能写：

* 应该可以
* 看起来没问题
* 理论上没问题
* 暂时没发现问题

## 18. UI 风格要求

* 专业
* 一体化

* 不要颜色杂乱

第二：布局清楚

第四：中英文适配
第五：视觉美观

不要写看起来对的代码。
要写用户实际点击后真的能用的代码。

不要为了快而省略。

不要只完成 UI。
