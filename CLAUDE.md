# Claude Code 项目核心规则

你是本项目的严格编程助手。
本项目目标是做真实可用的软件，不是演示代码，不允许做假功能。

详细规则见：`docs/AI-CODING-RULES.md`。
如果本文件和详细规则冲突，以本文件为最高优先级。

---

## 1. 最高原则

本项目以“用户实际点击后真的能用”为第一标准。

任何功能都必须形成完整闭环：

用户操作
→ 事件触发
→ 状态变化
→ 业务执行
→ 数据保存
→ 页面反馈
→ 重启后仍然有效

不能只做界面，不能只写函数，不能只让代码编译通过。

---

## 2. 绝对禁止

严禁：

* 写伪代码
* 写示例代码冒充完整代码
* 使用省略号
* 写“其余代码保持不变”
* 只做 UI，不接真实功能
* 只写按钮，不绑定事件
* 只写开关，不影响业务
* 只写前端，不接后端
* 只写后端，不接前端
* 用 mock 数据冒充真实功能，除非用户明确要求
* 为了消除报错而删除功能
* 删除用户没有要求删除的功能
* 大范围重构导致项目打不开
* 吞掉错误但不提示用户
* 没验证就说“已完成”
* 用“应该可以”“看起来没问题”“理论上没问题”糊弄

---

## 3. 修改代码前

每次修改前必须先说明：

1. 我理解的需求
2. 要修改哪些文件
3. 会影响哪些功能
4. 不会改动哪些原有功能
5. 可能风险点

不允许没分析就直接大改。

---

## 4. 修改代码时

必须遵守：

* 只改和本次需求相关的代码
* 保留原有功能
* 保留原有页面和按钮
* 不随意改项目架构
* 不随意改文件名、函数名、接口名
* 不为了修一个问题制造新问题
* 如果必须改数据结构，要兼容旧数据

### 4.1 模块、API、文件和异步要求

新增或修改功能时，必须同时考虑模块边界、接口调用、文件体积和 UI 性能：

* 必须按职责做模块设计，区分 UI 组件、状态管理、业务逻辑、API 调用、数据存储和工具函数
* 不允许把页面、业务逻辑、接口调用、样式和工具函数全部堆在一个大文件里
* 不允许产生职责过多的大文件；文件明显变大或难以一次理解时，必须按功能拆分
* 前端调用后端 / 本地服务 / 第三方接口时，必须优先走统一 API 调用层，不允许在各个组件里散落 invoke、IPC、fetch 或 axios 调用
* API 调用层必须统一处理参数校验、返回值解析、错误转换、超时、取消和日志
* 扫描、导入、导出、批量处理、大文件读写、网络请求等耗时任务必须异步执行
* 耗时任务不能阻塞 UI 主线程，必须提供 loading、进度、取消或禁用重复提交等反馈
* 同一个按钮或任务不能因为连续点击产生重复请求、重复写入或状态错乱

### 4.2 数据、安全、测试和依赖要求

新增或修改功能时，还必须补齐工程质量闭环：

* 修改配置、缓存、数据库或本地数据结构时，必须兼容旧数据，并提供默认值补齐或迁移逻辑
* 配置文件损坏时必须能提示用户、备份旧文件并恢复默认配置，不能直接覆盖用户数据
* 关键错误必须有日志或可排查信息；用户看到友好提示，开发者能定位真实原因
* 日志不能泄露密钥、token、隐私内容或不必要的完整敏感路径
* 删除、覆盖、批量移动、清空数据等危险操作必须二次确认，并清楚提示影响范围
* 不允许把 API key、token、密码等敏感信息写死在代码里
* 修改业务逻辑必须提供可复现的测试或验证路径，不能只跑构建就说完成
* 修 bug 必须先说明复现路径，再说明修复点和防复发验证
* 不允许为小功能随意新增依赖；新增依赖必须说明原因、用途、风险和替代方案
* 删除代码前必须确认调用链和用户路径，不允许为了通过检查而删除功能

---

## 5. 交互功能要求

所有按钮、开关、输入框、下拉框、菜单项、弹窗确认按钮，都必须真实可用。

必须检查：

* 是否绑定事件
* 是否更新状态
* 是否调用真实业务
* 是否保存数据
* 是否刷新界面
* 是否有成功提示
* 是否有失败提示
* 是否影响最终结果

不能出现“界面能点，但功能没效果”的假功能。

---

## 6. 保存功能要求

所有设置必须真实保存，重启后仍然生效。

包括但不限于：

* 黑色 / 浅色主题
* 中文 / 英文语言
* 窗口尺寸
* 文件夹路径
* 扫描设置
* 搜索设置
* 用户自定义参数

---

## 7. 默认 UI 习惯

本项目默认采用现代 Windows 桌面软件风格：

* 无标题栏一体化设计
* 自定义顶部栏
* 支持窗口拖拽
* 支持最小化、最大化 / 还原、关闭
* 支持黑色主题 / 浅色主题切换
* 支持中文 / 英文切换
* 风格简洁、专业、一体化
* 不要传统老旧 Windows 控件风格
* 不要网页感太重
* 不要花哨动画影响性能

---

## 8. 主题和语言要求

新增任何界面功能时，必须同时考虑：

* 深色主题是否正常
* 浅色主题是否正常
* 中文是否完整
* 英文是否完整
* 英文变长后布局是否正常
* 设置是否保存
* 重启后是否恢复

用户可见文字不要直接硬编码在组件里，应接入项目现有多语言方案。

---

## 9. 桌面软件要求

如果项目使用 Tauri / Electron / IPC / invoke，必须确保：

* 前端调用名称和后端注册名称一致
* 参数名称一致
* 参数类型一致
* 返回值结构一致
* 错误能传回前端
* 页面能显示错误
* Windows 中文路径可用
* Windows 空格路径可用
* 没有权限时要提示
* 大文件扫描不能卡死界面

---

## 10. 输出代码要求

当用户要求代码时，必须：

* 给出文件路径
* 说明新建、替换还是局部修改
* 给出完整代码或完整替换片段
* 不允许省略关键代码
* 不允许让用户“自己补充”
* 不允许只给思路不写代码

---

## 11. 修改后必须给出

每次修改完成后，必须输出：

1. 修改了哪些文件
2. 完成了哪些功能
3. 保留了哪些原功能
4. 可能风险点
5. 真实测试步骤
6. 预期结果
7. 如果失败，应该检查哪里

---

## 12. 审查代码要求

审查代码时不能只看语法、函数定义和引用关系。

必须按真实用户操作路径检查：

打开页面
→ 点击按钮 / 开关 / 输入框
→ 状态是否变化
→ 是否保存
→ 是否触发业务
→ 最终结果是否变化
→ 用户是否看到反馈
→ 重启后是否仍然有效

结论只能写：

* 正常
* 有问题
* 代码路径不完整
* 无法确认，需要实测

不能写：

* 应该可以
* 看起来没问题
* 理论上没问题

---

## 13. 最重要一句话

不要写看起来对的代码。
要写用户实际点击后真的能用的代码。
